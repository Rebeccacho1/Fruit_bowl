Brief

The database I will be making could potentially be suitable for libraries and book clubs. The main goal for this database is to allow members of the book club to access information about a book. This allows them to search the name of books which would give any information (like the name of the author, when it is published and so on). It gives them an opportunity to look for books they haven’t read based on genres and background information about the author. When we go to libraries, we often judge a book by the front cover and by making a database with only information (no front cover pictures) this helps eliminate the problem that is set around the world.

When I was trying to find answers to my queries and creating the databases, there were a few problems that I came across.

In the second screenshot (on the right), was another way of saying that I wanted the book_id and genre_id to be primary keys but in a different way. This time the database knew what I wanted to do.

In the first photo (the left one), I have created a problem which I thought was correct. Instead, the error had said “UNIQUE constraint failed: association_book_genre.book_id”.
Unique - being the only one of its kind; unlike anything else
Constraint - a limitation or restriction
The error above means: a problem which is one of its kind created a limitation. I decided to look into the problem further and it turns out that I had replaced a dot between the genre and book_id.
In the second photo (right one), there was only one small change which I had changed. This allows me to learn from my own mistakes by checking the code again before playing the sql code. The littlest mistakes can make the whole sql code incorrect even though I have done the rest of the code correctly.

The first screenshot wasn’t incorrect since it didn’t show a red box but it was incorrectly written which meant it didn’t give me the answer I wanted. When I used the code word “or” instead of “and” it will return with any title book written by Dan Brown or any book by Cathy Cassidy which is what I wanted. Relevant Implications Summary

End User Considerations

End User Considerations is about creating a database or any developed apps with the design decisions and presenting information to be helpful for the other end of the user rather than the developer. This allows the user to have a happy experience as it will be easy for them to use.
The database uses single word lower-case, field names and underscores to separate words instead of using spaces.
This is useful for the developer and necessary for the functionality of system (field names with spaces are more difficult to use in queries)
To address this relevant implication I have used aliasing with field names in the select part of the query.
This means that table titles in the data presentation are presented helpfully to the user.
For example page_count can be presented to the user as ‘Number of Pages’.

In the code above, you can see I have used aliases when there was a long title “select book_table.title” to a shortened title “titles” and it also worked on the title “book_table.author” to “authors”. This allows me to write a better title and make it better use (user friendly).


Sustainability and Future Proofing

Sustainability and Future Proofing is about making sure a digital product is easy to update and easy to add new features to.
To show Sustainability and Future Proofing, I have decided to Normalize my database because it is a method used widely everywhere to avoid any repetition and gaps in data sets. Normalisation is removing gaps and repeated data. In Version 1 - Stage 2 I decided to find awkwardly fitted data and it gave me an opportunity to Normalise my database. This means that any changes to the data can easily be updated because the update will only have to happen in one place. It also gives me an easier opportunity to add a new title without missing any of my other work up.
Since I have used relational tables, it will be easier to extend the database (which I will be doing in Version 3). In Version 3, I will be creating another simple table of readers who have read books. This will mean I just need to add an extra table and not having to make any changes to the data.


Cultural Implications

Cultural Implications is about making sure the digital product is culturally appropriate.
In New Zealand, there are many book titles that use the native New Zealand language (Māori). It is important that I made sure that they were stored correctly and appeared in the right form. I did tests to see if the unique letters appear.


Functionality

Functionality is the date of an app, website or database that is being useful to the user.
When I was creating the database table and finding the data type for each of the attributes, the ISBN numbers were a problem. If I put the ISBN numbers as the data type ‘integer’ it will chop off the starting zero. As this won’t be helpful because it has to have 13 numbers or 10 numbers.
For example: In the database, there is a book called ‘Ireland” which has the ISBN number of ‘0439014360’. As you can see there is 1 zero that will be chopped off if I put the data type as an integer.


If the ISBN number was put as an integer and there were dashes in between the numbers. It would also be problematic so I decided to make sure all of the ISBN had no dashes and were all consistent.
Another problem I found when I was collecting data was one of the books ‘Ngā Mahi Pārekareka kei te Marae, Fun at the Marae’ had an ISBN number that had no dashes while all the other books had dashes. To make life much simpler, it would be easier to take out all the dashes in the ISBN numbers.
Overall, I decided to put the ISBN number in text. This would mean the zeros wouldn’t be taken off but it would just mean the ISBN number wouldn’t be treated as a number as it wouldn’t be a problem throughout the process. Since I will not be adding or doing anything to the numbers.
